# 超长文档摄入策略：从 5 万字论文到一本书

> 综合业务约束、技术实现、成本模型，给出分层处理方案。  
> **现阶段只出方案，不改代码。**

---

## 一、先认清现状：当前系统对超长文档的处理

```
用户上传一本 30 万字的 PDF 书
        │
        ▼
┌─────────────────────────────────────────┐
│ 1. 文件接收 (app.py:1328)                │
│    无大小限制、无页数限制                 │
│    完整保存到 data/raw/papers/originals/  │
└─────────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────────┐
│ 2. 解析 (DeterministicParser.parse_pdf)  │
│    pdftotext -layout [file] -             │
│    timeout=60s                            │
│    30 万字的 PDF 可能输出 20 万字符        │
└─────────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────────┐
│ 3. 截断 (compiler.py:311)                │
│    source_text = source_content[:12000]   │
│    20 万字符 → 只剩 1.2 万字符            │
│    丢失率: 94%                            │
└─────────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────────┐
│ 4. LLM 编译                              │
│    prompt 里塞 1.2 万字符 + 全量 wiki 索引 │
│    output 上限 8000 tokens                │
│    试图从一本书的前 6% 提取所有知识        │
└─────────────────────────────────────────┘
```

### 现状的荒诞之处

- **磁盘上**：30 万字完整保存 ✅
- **Source Archive**：30 万字完整保存 ✅
- **LLM 看到的**：前 1.2 万字符（大概是前言+目录+第一章）
- **Knowledge Pages**：基于前 6% 的内容生成，后面的 94% 白读了 ❌

这不是"处理了但不够好"，这是**在假装处理**。

## 二、业务约束：我们到底在解决什么问题？

### 约束 1：产品定位 — Second Brain，不是文档仓库

SageMate 的核心价值是**从原始材料中提取结构化知识**。

如果用户上传一本书，他的真实意图可能是：
- **A. 我要精读这本书** → 他希望得到每章的 key takeaways、概念卡片
- **B. 我要查这本书** → 他只是想让这本书可被搜索，不急于提取知识页
- **C. 我要引用这本书** → 只需要一个 Source Archive + 几个核心概念页

不同意图对应不同的处理深度。**不应该用同一种方式处理所有文档。**

### 约束 2：成本现实 — 一本书的编译成本

以 qwen-plus 定价（$0.40/1M input, $1.20/1M output）估算：

| 处理方式 | Input Tokens | Output Tokens | 单次成本 | 质量 |
|----------|-------------|--------------|---------|------|
| 当前（截断到 12K 字符） | ~8K | ~3K | $0.006 | ⭐ 只读了前言 |
| 完整塞入（30 万字 ≈ 200K tokens） | ~210K | 8K（上限截断） | ~$0.10 | ⭐⭐ output 被截断 |
| 分 Chunk 编译（10 chunks × 3 万字） | ~200K（分散在多次调用） | ~30K | ~$0.12 | ⭐⭐⭐⭐ 完整覆盖 |
| 分层编译（大纲扫描 + 精选章节） | ~80K | ~15K | ~$0.05 | ⭐⭐⭐⭐⭐ 有策略地深读 |

**关键洞察：**
- 把 30 万字完整塞入一次调用，input 成本 ~$0.08，但 output 被 8000 上限锁死，**质量极差**
- 分 10 次 chunk 编译，总成本 ~$0.12，但**每章都被充分处理**
- 成本差距只有 2 倍，但质量差距是 10 倍

### 约束 3：时间现实 — 用户等得起多久？

| 文档类型 | 当前耗时 | 用户耐心 |
|---------|---------|---------|
| 1 篇公众号文章（3K 字） | 15~30s | ✅ 完全接受 |
| 1 篇论文（1 万字） | 30~60s | ✅ 可以接受 |
| 1 本书（30 万字） | 5~30 min（chunk 化） | ⚠️ 需要进度反馈 |
| 1 本书（30 万字）单调用 | 3~5 min（但只读了 6%） | ❌ 不可接受 |

用户可以接受"处理一本书需要 10 分钟"，但**不能接受"10 分钟后发现只处理了前言"**。

### 约束 4：输出上限 — 8000 tokens 能产出多少知识页？

一个中等质量的知识页约 500~800 tokens（含 frontmatter + content + tags + links）。

- 8000 output ÷ 600 = **最多 13 个知识页**
- 一本书通常值得提取 30~80 个概念
- **单调用架构天然不适合处理书**

## 三、分层处理策略

根据文档长度和类型，走不同的处理管道：

```
                    用户上传文档
                         │
                         ▼
              ┌─────────────────────┐
              │   文档长度检测       │
              │  (字符数 / 页数 /    │
              │   预估 token 数)     │
              └─────────────────────┘
                         │
        ┌────────────────┼────────────────┐
        │                │                │
   < 5K 字          5K ~ 50K 字       > 50K 字
        │                │                │
        ▼                ▼                ▼
   ┌─────────┐    ┌─────────────┐   ┌──────────────┐
   │ 快车道   │    │ 标准 Chunk   │   │ 深度编译     │
   │ Fastlane│    │ Pipeline     │   │ Deep Compile │
   └─────────┘    └─────────────┘   └──────────────┘
   单调用直接编译   分 chunk 并行编译    大纲扫描 + 
                   (2~5 chunks)       精选章节编译
   耗时: 15~30s    耗时: 1~3 min       耗时: 5~15 min
```

### Tier 1: Fastlane（< 5K 字）—— 单调用直接编译

适用：笔记、短文、单篇公众号、邮件

- 走当前 pipeline，source 不截断或截断阈值放宽到 15K
- 一次 LLM 调用产出 Source Archive + Knowledge Pages
- 预期产出 3~8 个知识页

### Tier 2: 标准 Chunk 管道（5K ~ 50K 字）—— 并行分片编译

适用：论文、技术文档、系列文章、深度报告

**处理流程：**

```
Step 1: 分片
    按语义边界（章节标题、页码标记 <!-- page=N -->）
    将文档切成 3~8K 字符的 chunks
    每个 chunk 保留上下文衔接（overlap 200 字）

Step 2: 并行编译
    每个 chunk 独立调用 LLM，生成：
    - chunk_local_pages[]（该 chunk 提取的知识页）
    - chunk_concepts[]（该 chunk 涉及的概念）
    
    并发控制：Semaphore(3)，避免打爆 API

Step 3: 合并去重
    收集所有 chunk 的产出，按 slug 去重
    同一概念在不同 chunk 中出现 → 合并内容，保留最早出处

Step 4: 全局 Source Archive
    用所有 chunk 的 chunk_concepts 汇总
    生成全书的 Source Archive（summary + key_takeaways + 概念列表）

Step 5: 写入
    Source Archive → wiki/sources/{slug}.md
    Knowledge Pages → wiki/{entities|concepts|analyses}/
```

**成本估算**（3 万字符论文，分 5 个 chunks）：
- 5 次 LLM 调用 × ($0.004 input + $0.0036 output) = **$0.038**
- 产出 15~25 个知识页（不再受 8000 output 限制）

### Tier 3: 深度编译（> 50K 字）—— 大纲扫描 + 精选章节

适用：书籍、大部头教材、长篇专著

**为什么不用简单 Chunk 化？**

因为 30 万字分 10 个 chunks 并行编译，虽然能覆盖全文，但：
1. 成本高（$0.12）
2. 产出太多低质量页面（每个 chunk 各自为政，缺乏全局视角）
3. 用户可能只关心书中的 3~5 个核心章节

**处理流程：**

```
Step 1: 大纲扫描（快速、低成本）
    LLM 调用 #1：只读前 2 万字 + 目录页
    输出：全书结构树
    {
      "title": "深度学习",
      "chapters": [
        {"index": 1, "title": "引言", "pages": [1,10], "importance": "low"},
        {"index": 2, "title": "线性代数基础", "pages": [11,45], "importance": "medium"},
        {"index": 3, "title": "反向传播算法", "pages": [46,120], "importance": "high"},
        ...
      ],
      "estimated_concepts": ["反向传播", "梯度下降", "神经网络", ...]
    }
    成本：~$0.01，耗时 5~10s

Step 2: 用户确认（可选，可默认自动）
    前端展示大纲，让用户选择：
    - "编译全部章节"（适合精读）
    - "只编译高重要性章节"（默认，适合快速了解）
    - "只保存原文，暂不编译"（适合先存档）

Step 3: 精选章节编译
    只对 importance >= medium 的章节走 Tier 2 Chunk 管道
    低重要性章节（前言、附录、参考文献）只纳入 Source Archive，不生成知识页

Step 4: 全局整合
    所有精选章节的产出汇总
    生成全书的 Source Archive
    建立跨章节的 wikilink（如第 3 章的"反向传播"链接到第 2 章的"链式法则"）
```

**成本估算**（30 万字书籍，10 章中精选 5 章核心章节）：
- 大纲扫描：$0.01
- 5 章 × 每章 3 万字 × 5 chunks = 25 次 LLM 调用
- 总成本：**~$0.18**
- 产出 40~60 个高质量知识页（而非 100+ 低质量页）

## 四、关键工程决策点

### 决策 1：要不要让用户选择处理方式？

| 策略 | 体验 | 工程复杂度 | 推荐度 |
|------|------|-----------|--------|
| **全自动（系统决定）** | 最简，但可能猜错意图 | 低 | ⭐⭐⭐ |
| **半自动（提示用户）** | 好，给用户掌控感 | 中 | ⭐⭐⭐⭐ |
| **全自动 + 事后调整** | 最好，先处理再让用户删减 | 高 | ⭐⭐⭐⭐⭐ |

推荐：**全自动做默认，但在前端展示"处理策略"和"预计耗时"**，让用户知道系统在做什么。

比如上传一本书后，底部状态栏显示：
> 📚 检测到长篇文档（32 万字 / 12 章）  
> 策略：大纲扫描 → 精选 6 章核心内容 → 预计 8 分钟  
> [全部编译] [只编译核心章节] [仅存档，暂不编译]

### 决策 2：Chunk 的边界怎么切？

| 切分策略 | 优点 | 缺点 |
|---------|------|------|
| **固定字符数**（每 8K 字符一刀） | 简单 | 可能切断句子、段落、表格 |
| **按章节标题**（Markdown H1/H2） | 语义完整 | 有的文档没标题；章节长度不均 |
| **按页码标记**（`<!-- page=N -->`） | 精准，可配页数 | 只有 PDF 有页码标记 |
| **语义切分**（按段落+overlap） | 最优雅 | 需要文本分割算法 |

推荐：**混合策略**
1. 优先按章节标题（H1/H2）切分
2. 如果某章 > 10K 字符，再按段落语义切分
3. 相邻 chunks 保留 200 字符 overlap，避免概念在边界处被切断

### 决策 3：Chunk 之间的知识页去重怎么做？

同一个概念（如"梯度下降"）可能在第 3 章和第 5 章都被提取。

方案：
```
Chunk A 产出: {slug: "gradient-descent", title: "梯度下降", content: "...基础定义..."}
Chunk B 产出: {slug: "gradient-descent", title: "梯度下降", content: "...变种算法..."}

合并策略：
1. 按 slug 去重，保留一个页面
2. 内容合并：以最早出现的版本为主体，后续版本作为"补充视角"合并到 Related 部分
3. 或者用一次轻量 LLM 调用做"内容融合"
```

### 决策 4：全局锁怎么办？

如果一本书分 25 个 chunks 并行编译，当前全局锁 `_compiler_lock` 会把它们全部串行，总耗时从 8 分钟变成 40 分钟。

所以 **Chunk 化编译的前提是必须先移除全局锁**，换成：
- 同 source_slug 的写操作加细粒度锁
- LLM 调用阶段无锁并行
- 最终合并写入阶段加锁

### 决策 5：Source Archive 的原文怎么处理？

一本书的 Source Archive 如果像现在一样全文保存，会是一个 30 万字的 markdown 文件。

问题：
- 磁盘 I/O 没问题
- 但加载到编辑器预览时会很卡
- 搜索时我们只索引 frontmatter，body 不进入 FTS5，这是好的

建议：**保持现状**，Source Archive 就是完整原文的存档。但在前端展示时：
- 默认只显示 frontmatter（概览+目录+概念链接）
- 点击"展开原文"才加载完整内容（或分页加载）

## 五、边界情况与降级策略

### 情况 1：用户明确说"只要存档，不编译"

- 只走 Step 0（保存原文）+ Step 1（生成 Source Archive 的 frontmatter，没有 knowledge pages）
- LLM 只做一次轻量调用生成 summary（甚至可以不用 LLM，用首段提取）
- 成本近乎为零

### 情况 2：PDF 解析失败 / 太大会卡死

当前 `pdftotext` 60s timeout，一本扫描版 PDF（几百页图片）可能：
- 输出为空（因为没 OCR）
- 或输出超大但全是乱码

**防御策略：**
- 文件大小上限：> 100MB 拒绝，提示"文件过大"
- 解析后字符数检测：如果 `len(source_content) < 500`，提示"未能提取到有效文本，可能是扫描版 PDF"
- 页数上限：> 1000 页的 PDF 走"仅存档"模式，不自动编译

### 情况 3：LLM API 中断或限流

Chunk 化编译的 25 次调用中，如果第 18 次失败了：
- 已完成的 17 个 chunks 结果保留
- 失败的 chunk 单独重试（最多 3 次）
- 最终合并时标注"第 X 章编译失败"，不阻塞整体流程

## 六、推荐的分阶段实施路径

### Phase 1：前置修复（必须先做）
1. 移除 `_compiler_lock` 全局锁 → 细粒度锁
2. 增加文件大小/页数检测，超长文档提示用户选择策略
3. `pdftotext` 放入线程池，避免阻塞 event loop

### Phase 2：Fastlane + 标准 Chunk（1~2 周）
1. 实现文档长度自动检测
2. < 5K 字：直接编译（放宽 source 截断到 15K）
3. 5K~50K 字：自动分 2~5 chunks，并行编译，合并去重
4. 前端显示 chunk 进度（"正在处理第 3/5 段..."）

### Phase 3：深度编译（2~4 周）
1. > 50K 字：先大纲扫描，再精选章节编译
2. 前端增加"处理策略"交互（全部/核心/仅存档）
3. Source Archive 增加目录结构 frontmatter（`chapters:` YAML 列表）

### Phase 4：质量优化（长期）
1. Chunk overlap 策略调优
2. 跨 chunk 概念去重的 LLM 融合
3. 书籍级别的跨章节 wikilink 自动补全

---

## 七、一句话总结

> **对于超长文档，不应该问"能不能处理"，而应该问"以什么策略、什么深度、什么成本来处理"。**
>
> 当前系统的问题不是"截断到 12K"这个阈值太小，而是**没有策略——无论上传什么都是一刀切**。一本笔记和一本书被同等对待，这是产品设计层面的缺失。
>
> 正确的做法是：**短文档直接编译，中文档分片并行，长文档先读大纲再精选深读。** 这样成本只增加 2~3 倍，但质量提升 10 倍，且用户体验从"假装处理了"变成"真的读懂了"。
